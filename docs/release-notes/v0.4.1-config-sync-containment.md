# MailMind v0.4.1 Config Sync Containment

**Release Date:** 2026-06-23  
**Base Version:** v0.4.0-job-experience  
**Branch:** fix/v041-config-sync-containment

## Summary

v0.4.1 is a stability and containment release that hardens local configuration loading, prevents duplicate sync jobs, and improves frontend job trigger UX. This release focuses on reliability and developer experience without introducing new features.

## Backend Changes

### Configuration Hardening
- **Local config loading**: Backend now loads settings from both `backend/.env` and `backend/.env.local`, with `.env.local` taking precedence.
- **Shared Settings object**: FastAPI and Celery worker now share the same Settings instance, ensuring consistent configuration across API and background jobs.
- **Path resolution**: `BACKEND_DIR` is now anchored independently of shell working directory, preventing config loading failures when starting services from different directories.

### Sync Job Containment
- **Duplicate job prevention**: `enqueue_sync_today_job` now returns an active queued/running job for the same user/mailbox instead of creating duplicates.
- **Redis sync lock**: Worker execution is guarded by a per-mailbox Redis lock (`sync:mailbox:{mailbox_id}`), preventing concurrent sync operations on the same mailbox.
- **Scheduled/manual mutual exclusion**: Scheduled email sync now calls the same enqueue path as manual sync, ensuring both paths respect the duplicate prevention logic.

### Error Handling and Retry
- **Network error classification**: Gmail timeout and TLS failures now map to stable network error codes for better diagnostics.
- **Retry with backoff**: Celery retries retryable sync failures with exponential backoff and jitter, up to 3 attempts.
- **Error redaction**: All error messages continue to be redacted to prevent secret leakage.

### Email Upsert Confirmation
- **Unique constraint preservation**: Email upsert behavior is confirmed to work correctly under the existing `(mailbox_id, external_id)` unique constraint.
- **No duplicate rows**: The combination of duplicate job prevention and unique constraints ensures no duplicate email rows are created.

## Frontend Changes

### Job Trigger Hardening
- **Mailbox sync button**: Disables when the mailbox has a queued/running active job, preventing repeated triggers.
- **Mailbox settings**: Restores active email sync jobs from recent jobs after page refresh, showing accurate job state.
- **Digest dashboard**: Disables Generate and Refresh buttons while a digest job is queued/running.
- **Digest state restoration**: Restores active digest jobs from recent jobs after page refresh.

### UX Improvements
- **Active job visibility**: Users can see when a job is already in progress and are prevented from accidentally triggering duplicates.
- **State persistence**: Job state survives page refresh, providing a more reliable experience.

## Development Experience

### Local Configuration
- **Separate .env.local files**: Both backend and frontend now support `.env.local` files for local configuration that won't be committed.
- **Git ignore**: `.env.local` files are properly ignored by Git to prevent accidental secret commits.
- **Example files**: Added `backend/.env.example` and `frontend/.env.example` to document required configuration.

### Development Scripts
- **PowerShell scripts**: Added `scripts/dev-backend.ps1`, `scripts/dev-worker.ps1`, `scripts/dev-frontend.ps1`, and `scripts/dev-all.ps1` for easier local development.
- **Consistent startup**: Scripts handle environment setup and service startup in a consistent way.

## Security Notes

- **No secret leakage**: All error messages continue to be redacted.
- **Config file security**: `.env.local` files are Git-ignored to prevent accidental commits.
- **No new attack surface**: This release does not introduce new API endpoints or authentication changes.

## Validation

- **Backend tests**: 198 tests passing (11 new tests for config loading, duplicate prevention, and error handling).
- **Frontend validation**: Typecheck, lint, and build all passing.
- **Secret scan**: No secrets found in tracked files or commit history.

## Known Limitations

- **No production scheduler**: Celery Beat and production-grade scheduling remain out of scope.
- **No historical cleanup**: Existing duplicate email rows (if any) are not automatically deleted.
- **Worker dependency**: If the Celery worker is offline, active jobs remain visible until they reach a terminal state or polling times out.
- **Backend authority**: Frontend duplicate prevention is UX-only; backend deduplication remains the authority.

## Migration Notes

- **No database migrations**: This release does not include any database schema changes.
- **No breaking changes**: All changes are backward compatible with v0.4.0.

## Upgrade Path

From v0.4.0:
1. Pull latest master.
2. Copy `backend/.env.example` to `backend/.env.local` and fill in local values.
3. Copy `frontend/.env.example` to `frontend/.env.local` and fill in local values.
4. Restart backend and Celery worker to pick up new config loading logic.
5. Restart frontend to pick up new job trigger hardening.
